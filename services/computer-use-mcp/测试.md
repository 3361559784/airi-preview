# `computer-use-mcp` 发版验收文档（Release Verification）

这份文档用于发版前验收，不是开发实现指南。目标不是“模型自报成功”，而是：改动能被别人复查、可重跑、可追责。

## 发布范围

- 适用于 `services/computer-use-mcp` 的代码变更发布前合规性检查。
- 适用对象：`coding` 相关改动、runner 变更、terminal/browser DOM 运行链路、approval / audit / trace 打分支路。

## 事实源与一致性

发版验收口径以以下事实源为准：

- `README.md`
- `agent.md`
- `src/support-matrix.ts`
- `services/computer-use-mcp/package.json` 的脚本定义
- 相关实现/测试文件

`README.md` 用作入口说明，不是唯一真值源；与 `support-matrix.ts`、`package.json scripts` 和实现状态不一致时，以后者为准。

## 发布结论口径

- 通过：满足下方所有“必满足项”，并产出可审计的机器可消费报告。
- 阻塞：任一必满足项未满足，或证据缺失。

## 一句话目标

每次改动完成后，必须能回答四个问题：

1. 改了什么？
2. 验证了什么？
3. 失败时会不会明确失败？
4. 证据能不能被别人重跑和复查？

缺任意一点都不算完成。

## 第一层：给所有人看的规则

### 1）完成的最小证明

每次改动都必须同时满足：

- 至少 1 条正向用例（成功路径）
- 至少 1 条负向用例（明确失败）
- 至少一个可复查证据（如 test output、run state、audit log、report）
- 在声明的环境约束内可重复执行，需报告环境前置条件
- 报告里必须保留“命令 / 退出码 / 关键字段”三件套

### 2）通过 / 失败定义

通过不等于“看到 ok”，必须同时满足：

- 命令 `exitCode === 0`
- 命令覆盖了本次改动范围
- 关键字段符合预期
- 无未处理 `unresolved issue`
- 无 `validation command mismatch`
- 无 `bad faith verification`（例如只跑了 `echo ok`、`pwd`、`ls`、`cat`、`true`、`exit 0`、`node -e`、`python -c`、`printf`，不能把它们当完成证明）
- 无 `non-zero terminal result` 被伪装成完成
- 证据写入可复查位置

失败只要命中任意一项即判定本次验收失败：

- 命令非 0 退出码
- 未运行验证命令
- 运行了错误验证命令
- 改了代码但验证未触及改动区域
- 负向用例未失败（静默成功）
- 报错不可读
- review 显示 ready 但 terminal evidence 实际失败
- scoped validation command 与实际命令不一致
- 未落地 report/state/audit/log 证据

> 失败后不是默认“回滚”，而是“不能把这次完成算通过”。  
> 回滚动作由阶段决策决定：  
> - 未提交改动：先修正。  
> - 已入主干且影响主流程：再评估 revert。  

### 3）三层验证顺序

1. 单元测试（unit）：先验证最小逻辑块
2. 联动测试（integration）：再验证模块协作是否打架
3. 整链路演练（E2E）：再验证从头到尾闭环

## 环境前置条件

验证报告必须声明至少以下项：

- OS / architecture
- Node.js 版本
- pnpm 版本
- commit hash
- 当前分支
- 是否启用 macOS executor
- 是否启用 browser DOM bridge
- 是否依赖 AIRI desktop
- 是否依赖真实浏览器 extension
- 输出目录

如果验证依赖本机权限、浏览器状态、AIRI desktop 或 macOS 窗口环境，必须标记：

- `environmentDependent: true`

## 第二层：发布验收命令矩阵

### 第 1 层：单元测试

适用场景：parser / guard / policy / verification gate / result shape / state transition / evidence builder 等。

最低要求：

- 至少 1 条正向、1 条负向用例
- 失败原因可读，不可静默
- 不依赖真实桌面、真实浏览器、真实终端

推荐命令：

```bash
pnpm -F @proj-airi/computer-use-mcp typecheck
pnpm -F @proj-airi/computer-use-mcp test
```

### 第 2 层：联动测试

适用场景：

- `coding-runner` 与 `verification-gate` 协作
- `terminal_exec` 与 terminal state 写入一致
- `coding_apply_patch` 后 review 串联
- `coding_report_status(completed)` 被 gate 拦截/放行
- approval / audit / trace 串联

最低要求：

- 一次成功路径
- 一次失败路径
- 失败不可静默通过
- 结果落到可复查对象

可复查对象（至少命中一个）：

- `CodingRunState`
- `lastTerminalResult`
- `lastChangeReview`
- `lastChangeDiagnosis`
- `verificationEvidenceSummary`
- `audit.jsonl`
- `report.json`

### 第 3 层：整链路演练

适用场景：coding workflow、terminal reroute、AIRI desktop、approval、browser DOM bridge。

推荐命令（按改动范围选）：

- `pnpm -F @proj-airi/computer-use-mcp e2e:coding-workflow`
- `pnpm -F @proj-airi/computer-use-mcp e2e:terminal-exec`
- `pnpm -F @proj-airi/computer-use-mcp e2e:browser-reroute`
- `pnpm -F @proj-airi/computer-use-mcp smoke:stdio`
- `pnpm -F @proj-airi/computer-use-mcp smoke:macos`

这不是每次改动都必须跑；改到 workflow、runner、approval、terminal、browser DOM bridge 时必须跑对应链路。

### 按改动范围选择验证命令

| 改动范围 | 最低验证 |
|---|---|
| 类型、schema、纯函数 | `typecheck` + `test` |
| `coding/*` | `test`（正负例必须覆盖） |
| `coding/verification-gate.ts` | gate 单测覆盖 pass / fail / recheck |
| `coding-runner/*` | runner 联动测试 + completion gate 测试 |
| `terminal/*` | terminal state 测试 + `e2e:terminal-exec` |
| `browser-dom/*` | DOM bridge 单测 + `e2e:browser-reroute` |
| `workflows/*` | workflow smoke / e2e |
| approval / audit / trace | 检查 `audit.jsonl`、`report.json` 落盘 |
| package/build 配置 | `typecheck` + `test` + 对应 smoke |

所选命令必须与 `support-matrix.ts` 的状态和 `package.json` 脚本保持一致，禁止用空跑命令替代关键验证动作。

### 每次改动最少记录

- 改动说明：改了什么、影响范围、不影响什么
- 验证说明：命令、exitCode、关键输出、证据路径
- 失败样例：错误输入、预期失败、实际错误、是否明确失败
- 结论：通过 / 不通过 / 部分通过 + 下一步优先修复点

## 第三层：给 agent / runner 的机器可消费报告

runner 的 JSON 报告必须固定字段，不靠自然语言说明。

### 必填字段

- `module`
- `changeScope[]`
- `gateDecision`（`pass` / `recheck_once` / `needs_follow_up` / `amend` / `abort`）
- `finalReportStatus`（`completed` / `in_progress` / `blocked` / `failed`）
- `workflowOutcome`（`completed` / `failed`）
- `commands[]`（每条含 `name` `command` `exitCode` `required` `coversChangeScope`）
- `evidence`（`terminalCommand` `terminalExitCode` `reportPath` `auditPath` `statePath`）
- `negativeCases[]`（每条含 `name` `expectedFailure` `observedFailure` `reasonCode`）
- `conclusion`（`accepted` `reason`）
- `environment`（`os` `node` `pnpm` `commit` `environmentDependent`）

### 字段注释（给执行者看）

- `module`：固定为 `@proj-airi/computer-use-mcp`。
- `changeScope`：本次改动实际触达的模块列表，用于判断命令是否覆盖。
- `environment`：至少包含执行环境和依赖的可复现信息。
- `gateDecision`：`evaluation` 层的主判定。
- `finalReportStatus`：run state 报告上可见的 `lastCodingReport.status` 语义。
- `workflowOutcome`：整个 coding workflow 是否成功落地。
- `commands[]`：所有验证命令。`required=true` 的命令必须成功；`coversChangeScope=true` 的命令必须真的命中改动范围。
- `evidence`：关键落地路径和关键证据；优先落 `report.json`、`audit.jsonl`、`run-state.json`。
- `negativeCases[]`：本次必须能“明确失败”的失败用例。
- `conclusion`：最终结论。`accepted=true` 表示可以接受为完成（前提条件满足）。

固定 JSON 示例：

```json
{
  "module": "@proj-airi/computer-use-mcp",
  "changeScope": ["coding", "coding-runner"],
  "environment": {
    "os": "macOS",
    "node": "unknown",
    "pnpm": "unknown",
    "commit": "unknown",
    "environmentDependent": false
  },
  "gateDecision": "pass",
  "finalReportStatus": "completed",
  "workflowOutcome": "completed",
  "commands": [
    {
      "name": "unit",
      "command": "pnpm -F @proj-airi/computer-use-mcp test",
      "exitCode": 0,
      "required": true,
      "coversChangeScope": true
    }
  ],
  "evidence": {
    "terminalCommand": "pnpm -F @proj-airi/computer-use-mcp test",
    "terminalExitCode": 0,
    "reportPath": "artifacts/computer-use-mcp/report.json",
    "auditPath": "artifacts/computer-use-mcp/audit.jsonl",
    "statePath": "artifacts/computer-use-mcp/run-state.json"
  },
  "negativeCases": [
    {
      "name": "reject bad-faith verification",
      "expectedFailure": true,
      "observedFailure": true,
      "reasonCode": "verification_bad_faith"
    }
  ],
  "conclusion": {
    "accepted": true,
    "reason": "Required commands passed, scope was covered, and negative case failed explicitly."
  }
}
```

### 判定映射要求

`pass` 判定需要 `gateDecision`、`finalReportStatus`、`workflowOutcome` 一致，并满足以下条件：

- `commands` 中 required 命令均 0 退出码
- 关键字段存在且符合预期
- `conclusion.accepted === true`
- 负向案例有明确失败
- `environmentDependent` 为 `true` 时，允许结果有环境波动，但必须完整写明前置条件

## 发布输出

每次发版验收必须附带一份结构化报告（JSON），并落盘到此次发版的证据目录。`commands`、`evidence`、`environment` 缺一不可。

## Azure 自动化（第二层，不替代本地）

Azure 跑分支只做固定复现，不替代本地基础验证。  
建议固定流程：

1. checkout 固定仓库
2. checkout 固定 commit
3. 安装固定依赖
4. 运行固定命令
5. 产出固定 JSON
6. 仅用 JSON 判定通过/不通过

## 发布落地动作

1. 在 `services/computer-use-mcp` 目录内执行发版前验收：
   - `pnpm -F @proj-airi/computer-use-mcp verify:local`
   - `pnpm -F @proj-airi/computer-use-mcp verify:coding`
   - `pnpm -F @proj-airi/computer-use-mcp verify:terminal`
   - `pnpm -F @proj-airi/computer-use-mcp verify:browser-dom`
2. 生成并提交结构化报告（`commands / evidence / environment / conclusion`）。
3. 在 `services/computer-use-mcp/package.json` 保持固定脚本不漂移（见脚本定义）。
   - `verify:local`：本地最小闭环  
   - `verify:coding`：coding 线路最小联动
   - `verify:terminal`：terminal 线路最小联动
   - `verify:browser-dom`：browser-dom 线路最小联动
4. PR/合并请求说明必须包含：
   - 改动范围（命中的模块）
   - 命令与 exitCode
   - `report.json` / `audit.jsonl` / `run-state.json` 证据路径
   - `environment` 字段是否 `environmentDependent` 与前置条件

## 不作为发版合规范围

不要求纳入本发版合规清单的内容（除非有明确新增风险）：

- 运行环境长期偏好优化
- 非本次改动触达区域的工具目录与检索体验重构
